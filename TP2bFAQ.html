Shared clarifications on students' FAQs (on-going TP2 or WA#2)

TP2 version b)

Obs:

Generic Questions
  • Q) Can we iterate the requirements with a possible extended specification beyond the basic requirements ?
      Yes. Stduents must use the initial requirements as a mandatory baseline and must try to address the suggested functionality and required security support as mandatory requirements to be addressed. However, students can use the initial statement (it is available in DOC) and extend the interpretation after possible clarifications in the LAB or in the available slots to discuss with the teacher.
  • Q) For the implementation of the TP2b there will be initial materials to be considered ?
      You can see bootstrap code for TP2a. Eventually you can use the same approach for TP2b. But this is only illustratitive implementation code. You can see it as reference for each implementation to address the requirements. For the TLS requirements (fine-tune security parameterizations) students must consider existent materials discussed in Lab 6. For cryptographic constructions that can be required, please take into account the LAB materials on cryptography and JAVA JCE programming.
  • Q) For a more specific approach, how is it supposed to address the parameterization of TLS endpoints for TLS protocol version, mutual authentication or enabled supported ciphersuites? ?
      In LAB 6 there are materials discussing how to have these parameterizations, as well as, debugging facilities. The Lab6 hands-on exercises are particularly focused in using different tools that you will possibly need for your implementation and debugging purposes: keytool, openssl, wireshark, ssldump or tls-tester.

      There are also code-exemples to start programming with Java JSSE. Anyway, as stated in the TP2 requirements, the implementation can be done with other programming paradigms and facilities, ex., REST/HTTPS (or REST/TLS), WebSockets/TLL or using specifically a WEB-APP Programming Platform (such as Spring or other).

  • Q) But how can I address the required parameterizations for clients and server endpoints to have a configurable implementation, as required ?
      A stating point is to consider that, beyond the necessary setup parameterizations for clients and server endpoints (certificates, certificate-chains, keystores, trustedstores, etc) you must have a TLS configuration file where you can express the initial TLS conditions for handshake, ex:

      File TLS.conf 

      <TLSVERSION>
tls-protocol-versiobs
</TLSVERSION>
<ENABLEDCIPHERSUITES>
Proposed-enabled-ciphersuites
</ENABLEDCIPHERSUITES>
<AUTHMODE>
authentication-mode
<AUTHMODE>
